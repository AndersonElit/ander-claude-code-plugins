## Event Envelope Standard

Every event must include a standard envelope wrapping the business payload. This ensures consistent handling across all services.

```json
{
  "eventId": "uuid-v4",
  "eventType": "orders.OrderPlaced",
  "version": "1.0.0",
  "timestamp": "2024-01-15T10:30:00.000Z",
  "correlationId": "uuid-v4",
  "causationId": "uuid-v4",
  "source": "ms-orders",
  "subject": "order-12345",
  "dataContentType": "application/json",
  "data": { }
}
```

| Field | Purpose | Generation Rule |
|-------|---------|----------------|
| `eventId` | Unique identifier for deduplication | UUID v4, generated by producer |
| `eventType` | Fully qualified event name | `<context>.<EventName>` |
| `version` | Schema version (semver) | Bumped on breaking/additive changes |
| `timestamp` | When the event occurred | ISO 8601 UTC, set by producer |
| `correlationId` | Links all events in a business flow | Propagated from the initiating request |
| `causationId` | The event that directly caused this one | eventId of the triggering event |
| `source` | Service that produced the event | Service name or URN |
| `subject` | The aggregate/entity this event relates to | Entity ID or composite key |
| `dataContentType` | Media type of the data payload | Usually `application/json` |
| `data` | The business payload | Event-specific schema |

This structure follows CloudEvents specification conventions, which makes the events portable across different brokers and tooling.

## JSON Schema Templates

### Domain Event Schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://schemas.example.com/orders/order-placed/v1.0.0",
  "title": "OrderPlaced",
  "description": "Emitted when a customer successfully places a new order",
  "type": "object",
  "required": ["eventId", "eventType", "version", "timestamp", "correlationId", "source", "data"],
  "properties": {
    "eventId": { "type": "string", "format": "uuid" },
    "eventType": { "type": "string", "const": "orders.OrderPlaced" },
    "version": { "type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$" },
    "timestamp": { "type": "string", "format": "date-time" },
    "correlationId": { "type": "string", "format": "uuid" },
    "causationId": { "type": "string", "format": "uuid" },
    "source": { "type": "string" },
    "subject": { "type": "string" },
    "dataContentType": { "type": "string", "default": "application/json" },
    "data": {
      "type": "object",
      "required": ["orderId", "customerId", "items", "totalAmount", "currency"],
      "properties": {
        "orderId": { "type": "string", "format": "uuid" },
        "customerId": { "type": "string", "format": "uuid" },
        "items": {
          "type": "array",
          "minItems": 1,
          "items": {
            "type": "object",
            "required": ["productId", "quantity", "unitPrice"],
            "properties": {
              "productId": { "type": "string" },
              "quantity": { "type": "integer", "minimum": 1 },
              "unitPrice": { "type": "number", "minimum": 0 }
            }
          }
        },
        "totalAmount": { "type": "number", "minimum": 0 },
        "currency": { "type": "string", "pattern": "^[A-Z]{3}$" },
        "shippingAddress": {
          "type": "object",
          "properties": {
            "street": { "type": "string" },
            "city": { "type": "string" },
            "country": { "type": "string", "pattern": "^[A-Z]{2}$" },
            "postalCode": { "type": "string" }
          }
        }
      },
      "additionalProperties": false
    }
  }
}
```

## Versioning Strategy

Treat event schemas the same way you treat public APIs — with semantic versioning and backward compatibility.

### Version Bump Rules

| Change Type | Example | Version Bump | Compatibility |
|-------------|---------|-------------|---------------|
| Add optional field | Add `discountCode` to OrderPlaced | MINOR (1.0 → 1.1) | Backward compatible |
| Add required field | Add `warehouseId` to StockReserved | MAJOR (1.x → 2.0) | Breaking |
| Remove field | Remove `legacyCode` | MAJOR | Breaking |
| Rename field | `amount` → `totalAmount` | MAJOR | Breaking |
| Change field type | `quantity: string` → `quantity: int` | MAJOR | Breaking |
| Add enum value | Add `CRYPTO` to PaymentMethod | MINOR | Backward compatible (if consumers handle unknown) |
| Remove enum value | Remove `CHECK` from PaymentMethod | MAJOR | Breaking |

### Multi-Version Support Strategy

When you publish a breaking change:

1. **Publish both versions simultaneously** on separate topics or with version routing:
   - `orders.order-placed.v1` and `orders.order-placed.v2`
   - Or use a `version` header for topic-level routing
2. **Set a deprecation timeline** — e.g., v1 will be removed in 90 days
3. **Monitor consumer migration** — track which consumers still read v1
4. **Sunset the old version** only when all consumers have migrated

### Schema Evolution with Avro

If using a Schema Registry (e.g., Confluent):

| Compatibility Mode | What You Can Do | What You Cannot Do |
|-------------------|----------------|-------------------|
| **BACKWARD** (recommended) | Add optional fields, remove fields | Add required fields |
| **FORWARD** | Add fields (optional or required), remove optional fields | Remove required fields |
| **FULL** | Add/remove optional fields only | Add/remove required fields |
| **NONE** | Anything | — (no guarantees) |

Use **BACKWARD** compatibility as the default — it ensures new consumers can read old events.

## Compatibility Rules

### Self-Contained Events

Events should carry enough information for consumers to act without calling back to the producer. This is the "fat event" approach:

**Good** — self-contained:
```json
{
  "eventType": "orders.OrderPlaced",
  "data": {
    "orderId": "abc-123",
    "customerId": "cust-456",
    "customerEmail": "user@example.com",
    "items": [
      { "productId": "prod-789", "productName": "Widget", "quantity": 2, "unitPrice": 29.99 }
    ],
    "totalAmount": 59.98,
    "currency": "USD",
    "shippingAddress": { "city": "Madrid", "country": "ES" }
  }
}
```

**Avoid** — requires callback:
```json
{
  "eventType": "orders.OrderPlaced",
  "data": {
    "orderId": "abc-123"
  }
}
```

The trade-off: fat events increase message size but eliminate temporal coupling (what if the producer is down when the consumer tries to fetch details?).

### Sensitive Data in Events

Never include PII or credentials in plaintext events. Options:

- **Field-level encryption**: Encrypt sensitive fields, provide decryption keys only to authorized consumers
- **Reference pattern**: Store sensitive data in a secure vault, include a reference token in the event
- **Data classification**: Tag fields with sensitivity levels in the schema, enforce handling policies per level

## Consumer-Driven Contracts

Let consumers define what they need from a producer, rather than the producer guessing what matters.

### How It Works

1. **Consumer writes a contract** specifying the fields and values it expects:
   ```json
   {
     "consumer": "ms-notifications",
     "provider": "ms-orders",
     "event": "orders.OrderPlaced",
     "expectations": {
       "data.orderId": { "type": "string", "required": true },
       "data.customerEmail": { "type": "string", "format": "email", "required": true },
       "data.items": { "type": "array", "minItems": 1, "required": true }
     }
   }
   ```

2. **Producer verifies contracts in CI** — before publishing a schema change, run all consumer contracts against the new schema. If any break, the build fails.

3. **Contract testing tools**: Pact (most popular), Spring Cloud Contract, custom JSON Schema validation in CI pipeline.

### Why This Matters

Without consumer-driven contracts, a producer might remove a field thinking nobody uses it — and silently break three downstream services. Contracts make these dependencies explicit and enforce them automatically.
